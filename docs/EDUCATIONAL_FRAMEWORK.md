# AWS AI 應用架構師培養框架

**目標:** 培養能夠獨立設計和架設完整 AI 應用的架構師

**核心原則:** 不只教「怎麼做」，更要教「為什麼」——每個架構決策都有充分的理由

---

## 教學哲學

### 1. 架構思維 vs. 實作技能

| 實作技能 | 架構思維 |
|----------|----------|
| 怎麼寫 Terraform | 為什麼要用 Terraform |
| 怎麼建 VPC | 為什麼需要 VPC |
| 怎麼配置 Lambda | 為什麼選 Serverless |
| 怎麼部署服務 | 為什麼這樣部署 |

**兩者都重要，但架構思維決定了系統的上限**

### 2. 每個技術決策必須回答 4 個問題

1. **為什麼需要這個組件？**
   - 解決什麼問題？
   - 不用它會怎樣？

2. **為什麼選這個方案？**
   - 有哪些替代方案？
   - 比較後的優缺點？

3. **這個方案的限制是什麼？**
   - 什麼情況下不適用？
   - 擴展性如何？

4. **什麼時候需要重新考慮？**
   - 流量增長 10 倍？
   - 需要多區部署？
   - 合規要求變更？

## 重新設計的 Stage 1 教學架構

### 模組 1：為什麼需要雲端網路基礎？

#### 1.1 從問題出發

**場景：** 你要建置一個 AI 應用，需要提供 API 給客戶使用

**問題 1：** 為什麼不能直接把伺服器放在網路上？

```
傳統方式：
[用戶] <---> [暴露在公網的資料庫/API伺服器]
               ⚠️ 安全風險極高
```

**雲端方式：**
```
[用戶] <---> [負載均衡器] <---> [公有子網的 API] <---> [私有子網的資料庫]
                           ↑                    ↑
                      [安全防護層]         [受保護的核心資產]
```

**核心概念：防禦深度 (Defense in Depth)**
- 多層防禦：每層都有安全控制
- 最小權限：只開放必要的通訊
- 隔離：不同層級的資源分開部署

#### 1.2 VPC 是什麼？為什麼需要它？

**VPC (Virtual Private Cloud) = 你在 AWS 雲端的私人網路**

**類比理解：**
```
AWS 雲端 = 一個巨大的辦公大樓
VPC = 你租的一層樓（有自己的網路規劃）
子網 = 樓層內的辦公室區域
安全群組 = 每個辦公室的門禁系統
```

**為什麼需要 VPC？**

| 原因 | 說明 |
|------|------|
| **安全隔離** | 你的資源與其他 AWS 用戶隔離 |
| **網路控制** | 完全控制 IP 範圍、路由規則 |
| **合規要求** | 許多產業法規要求数據隔離 |
| **成本管理** | 可以使用專屬連線降低成本 |

**不使用 VPC 的後果：**
- 所有資源在預設的 AWS 帳號網路
- 無法做細粒度的網路控制
- 安全性完全依賴個別資源的安全設定
- 不符合企業合規標準

### 模組 2：網路架構設計決策

#### 2.1 公有子網 vs. 私有子網

**核心問題：** 哪些資源該放公有子網？哪些該放私有子網？

**決策樹：**

```
這個資源需要直接對外服務嗎？
├─ 是 → 公有子網 (API Server、Load Balancer)
└─ 否 → 私有子網 (資料庫、內部服務)
    │
    需要對外存取資料嗎？
    ├─ 是 → 私有子網 + NAT Gateway
    └─ 否 → 完全私有子網 (無外部連線)
```

**具體案例：AI 應用的資源分配**

| 資源類型 | 放置位置 | 原因 |
|----------|----------|------|
| API Gateway | 公有子網 | 需要接收外部請求 |
| Lambda (API 處理) | 公有子網 | 無狀態，處理請求 |
| EC2 (AI 模型服務) | 私有子網 | 昂貴資源，需要保護 |
| RDS (向量資料庫) | 私有子網 | 數據庫必須在私有網路 |
| S3 Gateway Endpoint | 私有子網 | 安全存取儲存 |

**為什麼這樣分配？**

**公有子網的優點：**
- ✅ 直接對外服務，架構簡單
- ✅ 客戶訪問延遲最低
- ✅ 運維相對簡單

**公有子網的缺點：**
- ❌ 安全風險較高
- ❌ 受攻擊面積大
- ❌ 不適合放敏感資料

**私有子網的優點：**
- ✅ 安全性高，不直接暴露
- ✅ 可以使用專屬硬體
- ✅ 符合合規要求

**私有子網的缺點：**
- ❌ 需要 NAT Gateway (成本 ~$32/月)
- ❌ 架構複雜度增加
- ❌ 排查問題較困難

#### 2.2 NAT Gateway 的權衡

**問題：** 私有子網的伺服器需要對外存取（例如呼叫 AI API），該怎麼做？

**方案 A：NAT Gateway** ✅ 選擇

```
[私有子網 EC2] → [NAT Gateway] → [Internet Gateway] → [外部 API]
                       ↑
                   (有 EIP 的公網 IP)
```

**優點：**
- ✅ AWS 託管服務，高可用
- ✅ 自動擴展，無需維護
- ✅ 與 AWS 服務整合良好

**缺點：**
- ❌ 固定成本 ~$32/月（不論流量）
- ❌ 單點故障風險（每 AZ 一個）
- ❌ 處理大量流量時昂貴

**適用場景：**
- 生產環境
- 需要高可用性
- 流量不穩定但不大

---

**方案 B：NAT Instance**

```
[私有子網 EC2] → [NAT Instance (自行管理)] → [Internet Gateway] → [外部 API]
```

**優點：**
- ✅ 成本較低（$5-15/月）
- ✅ 可以完全控制
- ✅ 可以安裝額外軟體

**缺點：**
- ❌ 需要自己維護和更新
- ❌ 需要自己處理高可用
- ❌ 成為瓶頸的風險

**適用場景：**
- 開發/測試環境
- 成本敏感的專案
- 有運維團隊的情況

---

**方案 C：VPC Endpoint** (適用於 AWS 服務)

```
[私有子網 EC2] → [VPC Endpoint] → [AWS 服務，如 S3/DynamoDB]
                       ↑
                   (不經過公網)
```

**優點：**
- ✅ 不走公網，更安全
- ✅ 延遲更低
- ✅ 可能有成本優勢

**缺點：**
- ❌ 只能用於 AWS 服務
- ❌ 需要額外配置

**適用場景：**
- 需要存取 S3、DynamoDB 等
- 對安全性要求高
- 想降低成本

**我們 Stage 1 的選擇：NAT Gateway**
- 適合學習完整架構
- 生產環境常用方案
- 雖然有成本，但值得學習

#### 2.3 單 AZ vs. 多 AZ 部署

**問題：** 為什麼要跨多個 Availability Zone 部署？

**單 AZ 的問題：**

```
[AZ-a]
├─ 所有資源
└─ ❌ AZ 故障 = 所有服務中斷
```

**多 AZ 的優勢：**

```
[AZ-a]  [AZ-b]
  ↓        ↓
[負載均分] ← 故障轉移
  ↓
[服務持續可用]
```

**優點：**
- ✅ 高可用性：一個 AZ 故障不影響服務
- ✅ 災難恢復：區域級災難的保護
- ✅ 合規：許多產業要求多 AZ

**缺點：**
- ❌ 成本增加：資源翻倍
- ❌ 複雜度增加：需要跨 AZ 通訊
- ❌ 數據一致性：跨 AZ 資料同步

**什麼時候用多 AZ？**

| 情境 | 單 AZ | 多 AZ |
|------|-------|-------|
| 開發環境 | ✅ 可接受 | ❌ 過度設計 |
| 測試環境 | ✅ 可接受 | ❌ 成本不划算 |
| 生產環境 | ❌ 風險太高 | ✅ 必需 |
| 關鍵任務系統 | ❌ 不可接受 | ✅ 必需 |

**我們 Stage 1 的選擇：2 AZ**
- 學習生產級架構
- 了解高可用設計
- 成本可控（只用小實例）

### 模組 3：安全設計原則

#### 3.1 為什麼要分層防禦？

**單層防禦的問題：**
```
[外部攻擊] → [單一安全防線] → [你的資料]
                  ↑
         (這層被突破 = 全軍覆沒)
```

**多層防禦：**
```
[外部攻擊] → [WAF] → [Security Group] → [NACL] → [你的資料]
              ↑        ↑               ↑
         Web防護    實例級防護      子網級防護
```

**每層的作用：**

1. **WAF (Web Application Firewall)**
   - 過濾惡意流量（SQL Injection、XSS）
   - DDoS 防護
   - 應用層防護

2. **Security Group (實例級)**
   - 允許/拒絕特定 IP 和端口
   - 狀態防火牆（可以追蹤連線）
   - 最細粒度的控制

3. **NACL (子網級)**
   - 子網層級的規則
   - 無狀態防火牆
   - 補充安全群組

#### 3.2 最小權限原則

**核心概念：** 只給予必要的最小權限

**錯誤示例：**
```hcl
# ❌ 開放所有通訊
resource "aws_security_group" "web" {
  ingress {
    from_port   = 0
    to_port     = 65535
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
}
```

**正確示例：**
```hcl
# ✅ 只開放必要的端口
resource "aws_security_group" "web" {
  ingress {
    from_port   = 443
    to_port     = 443
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  ingress {
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = ["YOUR.OFFICE.IP/32"]  # 只允許辦公室 IP
  }

  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
}
```

**為什麼要這樣？**

| 原則 | 好處 |
|------|------|
| 只開放必要端口 | 攻擊面積最小 |
| 限制來源 IP | 降低暴力破解風險 |
| 預設拒絕 | 安全錯誤設計 |
| 明確記錄 | 可審計性 |

### 模組 4：成本考量

#### 4.1 Stage 1 成本分析

**固定成本（每月）：**

| 資源 | 成本 | 必要性 | 節省方案 |
|------|------|--------|----------|
| NAT Gateway | ~$32 | 必需（如需私網上網） | 開發時可停用 |
| Elastic IP (NAT 用) | $0 + 流量 | 必需 | - |
| **小計** | **~$32** | - | - |

**變動成本（每月，假設 24/7 運行）：**

| 資源 | 數量 | 單價 | 小計 | 必要性 |
|------|------|------|------|--------|
| EC2 t3.micro | 2 | ~$7.5 | ~$15 | 學習用可選 |
| 資料傳輸 | - | 變動 | ~$2 | - |
| **小計** | - | - | **~$17-20** | - |

**總計：~$50/月（如果 24/7 運行）**

**成本優化建議：**

1. **不用時停用資源**
   ```bash
   terraform destroy  # 完全移除
   # 或只停用 EC2（保留 NAT）
   ```

2. **使用 Spot/Reserved 實例**
   - 實驗性專案用 Spot（省 70-90%）
   - 長期專案用 Reserved（省 30-50%）

3. **單 AZ 部署（開發環境）**
   - 節省一半資源
   - 適合開發/測試

#### 4.2 成本 vs. 架構的權衡

```
           成本
            ↑
            │     [單 AZ]
            │    /
            │   /
      適合 │  /
      架構 │ /
            │/
            └──────────→ 可靠性

      [多 AZ]  ← 高成本高可靠性
```

**建議：**
- 開發：單 AZ，最簡架構
- 測試：多 AZ，小實例
- 生產：多 AZ，合適大小
