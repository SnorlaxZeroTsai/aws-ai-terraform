# 為什麼需要 Serverless？

**學習目標：** 理解 Serverless 架構的價值，知道什麼時候該用（或不該用）

---

## 傳統架構的痛點

### 場景：你的 AI 應用上線了

```
第 1 週：10 個用戶
→ 你部署了 1 台 EC2 (t3.medium)
→ 成本：$30/月
→ CPU 使用率：5%

第 4 週：100 個用戶
→ 流量增長 10 倍
→ 你擴展到 3 台 EC2
→ 成本：$90/月
→ CPU 使用率：15%

第 3 個月：1000 個用戶
→ 流量再增長 10 倍
→ 你擴展到 10 台 EC2
→ 成本：$300/月
→ CPU 使用率：20%

問題：
→ 80% 的時間伺服器都在閒置
→ 你為沒使用的容量付錢
→ 運維負擔越來越重
```

**這就是傳統架構的問題：**
- ❌ 必須預先準備容量
- ❌ 為閒置時間付費
- ❌ 手動管理擴展
- ❌ 運維負擔重

---

## Serverless 的解決方案

### 什麼是 Serverless？

**Serverless ≠ 沒有伺服器**

```
Serverless 的真實含義：
├─ 伺服器仍然存在（AWS 管理）
├─ 你看不到伺服器
├─ 你不管理伺服器
└─ 你只為實際使用付費
```

**類比：**

```
傳統架構（EC2）：
[你]
  ├─ 買車
  ├─ 保養車
  ├─ 加油
  └─ 停車

問題：車子閒置時也要付錢

Serverless（Lambda）：
[你]
  └─ 用 Uber

優點：
→ 只為搭乘付費
→ 不用管車的維護
→ 自動擴展（需要時叫車）
```

## 真實成本對比

### 場景 1：間歇性 API 服務

**需求：**
- 每天 1,000 次請求
- 每個請求處理 2 秒
- 只在工作時間（8 小時）使用

**傳統 EC2：**
```
EC2 (t3.medium): $30/月
→ 不管有沒有請求，都付 $30
→ 24 小時運行
→ 閒置時間：16 小時/天（67%）
→ 浪費：$20/月
```

**Serverless Lambda：**
```
Lambda 計算：
- 請求數：1,000 × 30 天 = 30,000 請求/月
- 執行時間：30,000 × 2s = 60,000 秒
- 記憶體：512MB

成本：
- 請求費：30,000 × $0.20 / 1M = $0.006
- 執行費：60,000s × 0.5GB × $0.00001667 = $0.50
- 總計：~$0.51/月

節省：$30 - $0.51 = $29.49/月（98% 節省）
```

### 場景 2：持續高流量服務

**需求：**
- 每秒 10 個請求（24/7）
- 每個請求處理 1 秒
- 需要快速響應

**傳統 EC2：**
```
需要 10 台 EC2（24/7 運行）
成本：10 × $30 = $300/月
```

**Serverless Lambda：**
```
Lambda 計算：
- 每月請求：10 × 60 × 60 × 24 × 30 = 25.9M 請求
- 執行時間：25.9M × 1s = 25.9M 秒
- 記憶體：512MB

成本：
- 請求費：25.9M × $0.20 / 1M = $5.18
- 執行費：25.9M s × 0.5GB × $0.00001667 = $216
- 總計：~$221/月

節省：$300 - $221 = $79/月（26% 節省）
```

**關鍵洞察：**
- 間歇性流量 → Serverless 節省最多（98%）
- 持續流量 → Serverless 仍能節省（26%）
- 流量越穩定，Serverless 優勢越小

## Serverless 的核心優勢

### 1. 零運維負擔

```
傳統 EC2：
[你負責]
├─ OS 更新
├─ 安全修補
├─ 容量規劃
├─ 負載均衡
├─ 健康檢查
├─ 日誌管理
├─ 監控設置
└─ 故障排除

Serverless Lambda：
[AWS 負責]
├─ OS 更新 ✓
├─ 安全修補 ✓
├─ 容量規劃 ✓
└─ [你只關注]業務邏輯
```

### 2. 自動擴展

```
傳統架構：
[流量暴增 10 倍]
  ↓
[手動偵測]
  ↓
[手動增加 EC2]
  ↓
[等待 5-10 分鐘]
  ↓
[可能已經超時]
```

```
Serverless 架構：
[流量暴增 10 倍]
  ↓
[AWS 自動擴展]
  ↓
[毫秒級響應]
  ↓
[自動收縮]
```

**實際案例：**
```
某電商促銷活動：
- 平時：100 請求/分鐘
- 促銷：100,000 請求/分鐘（1000 倍）

傳統 EC2：
→ 需要預先準備 1000 個實例
→ 或臨時手動擴展（可能來不及）
→ 成本：$30,000/月（準備容量）

Serverless Lambda：
→ 自動處理 1000 倍流量
→ 只為實際使用付費
→ 成本：根據實際請求（可能更低）
```

### 3. 按使用付費

```
傳統模式：
[購買容量]
├─ 預購 1 年：省 50%
├─ 預購 3 年：省 70%
└─ 風險：買多了浪費，買少了不夠

Serverless 模式：
[按需付費]
├─ 用多少付多少
├─ 無需預測
└─ 零風險
```

**成本預測：**
```
傳統 EC2：
→ 需要容量規劃
→ 預測錯誤 = 成本問題
→ 估計：±50% 誤差

Serverless Lambda：
→ 直接根據請求計算
→ 請求數 × 單價 = 總成本
→ 誤差：<5%
```

## Serverless 的限制

### 1. 執行時間限制

**Lambda 限制：**
```
最大執行時間：15 分鐘

適合：
✅ HTTP API 請求（秒級）
✅ 資料處理（分鐘級）
✅ 非同步任務（分鐘級）

不適合：
❌ 長時間批次處理（小時級）
❌ 影片編碼（小時級）
❌ 機器學習訓練（天級）
```

**解決方案：**
```
長時間任務：
[ Lambda ] → 啟動 → [ ECS Fargate ]
                  (觸發長時間任務)
```

### 2. 冷啟動延遲

**什麼是冷啟動？**
```
第一次請求：
→ AWS 準備執行環境
→ 載入你的程式碼
→ 初始化依賴
→ 開始執行

延遲：100-500ms（冷啟動）
後續請求：<10ms（熱啟動）
```

**冷啟動的影響：**
```
場景 1：低流量 API
- 每分鐘 1 個請求
- 每次都是冷啟動
- 延遲：500ms
- 用戶體驗：差

場景 2：高流量 API
- 每秒 100 個請求
- 大部分是熱啟動
- 延遲：10ms
- 用戶體驗：好
```

**優化策略：**
```
1. 預留並發（Provisioned Concurrency）
   → 保持函數熱狀態
   → 成本：$0.0000042333/GB-second
   → 適合：需要低延遲的場景

2. 優化初始化
   → 減少啟動時的載入
   → 使用輕量依賴
   → 延遲載入資源

3. 選擇合適語言
   → Node.js/Python：快（~100ms）
   → Java/Go：慢（~500ms）
```

### 3. 狀態管理

**問題：**
```
Lambda 每次執行都是新環境
→ 無法儲存本地狀態
→ 無法使用本地檔案系統
→ 無法保持連線
```

**解決方案：**
```
[Lambda 1]
    ↓ 寫入
[Amazon S3 / DynamoDB / EFS]
    ↓ 讀取
[Lambda 2]

外部狀態儲存：
→ S3：檔案儲存
→ DynamoDB：NoSQL 資料庫
→ EFS：檔案系統（可掛載到 Lambda）
```

## 什麼時候該用 Serverless？

### ✅ 適合的場景

**1. 間歇性工作負載**
```
特徵：
→ 流量波動大
→ 有明顯的峰值和低谷
→ 不是 24/7 運行

例子：
→ 辦公時間的 API
→ 批次處理任務
→ 定期排程任務
→ 事件驅動的處理
```

**2. 不可預測的流量**
```
特徵：
→ 無法預測請求量
→ 可能有突發流量
→ 需要自動擴展

例子：
→ 社群媒體整合
→ Webhook 處理
→ API 服務（成長期）
→ 促銷活動
```

**3. 微服務架構**
```
特徵：
→ 小型、獨立的服務
→ 透過 API 通訊
→ 快速部署需求

例子：
→ 認證服務
→ 通知服務
→ 圖片處理
→ 資料轉換
```

**4. 快速原型開發**
```
特徵：
→ 需要快速驗證想法
→ 無運維團隊
→ 預算有限

例子：
→ MVP 產品
→ 概念驗證
→ 學習專案
```

### ❌ 不適合的場景

**1. 長時間運行的任務**
```
特徵：
→ 執行時間 > 15 分鐘
→ 需要持續運行

例子：
❌ 影片編碼（大檔案）
❌ 機器學習訓練
❌ 大型資料處理
❌ 遊戲伺服器

替代方案：
→ ECS Fargate
→ EC2
→ SageMaker
```

**2. 高頻率、短延遲需求**
```
特徵：
→ 每秒數千請求
→ 需要 <10ms 延遲

例子：
❌ 高頻交易
❌ 實時遊戲
❌ IoT 裝置通訊

替代方案：
→ EC2（固定容量）
→ Alchemy（預留實例）
```

**3. 需要特殊環境**
```
特徵：
→ 需要 OS 級別控制
→ 特殊驅動程式
→ 複雜的依賴

例子：
❌ GPU 運算
❌ 特殊硬體整合
❌ 複雜的網路設定

替代方案：
→ EC2（完全控制）
→ ECS（部分控制）
```

**4. 持續高流量（> 70% 使用率）**
```
特徵：
→ 幾乎 24/7 滿載
→ 穩定的工作負載

例子：
❌ 生產資料庫
❌ 大型 API 服務

替代方案：
→ EC2（預留實例更便宜）
→ ECS（更好的資源利用率）
```

## 決策框架

### 選擇樹

```
你的工作負載是什麼？

[執行時間 > 15 分鐘？]
├─ Yes → ❌ 不要用 Lambda
│          → 用 ECS Fargate 或 EC2
└─ No → 繼續
    [需要 <10ms 延遲？]
    ├─ Yes → ❌ 可能不適合
    │          → 用 EC2 或考慮優化
    └─ No → 繼續
        [流量穩定且 24/7 滿載？]
        ├─ Yes → ⚠️ 考慮 EC2（可能更便宜）
        └─ No → ✅ Serverless 完美
            [間歇性或不可預測流量？]
            ├─ Yes → ✅ Serverless 理想
            └─ No → 考慮成本對比
```

### 成本轉換點

```
使用率 vs 成本對比：

0-20% 使用率：
→ Serverless: $10
→ EC2: $30
→ 節省：67%

20-50% 使用率：
→ Serverless: $25
→ EC2: $30
→ 節省：17%

50-70% 使用率：
→ Serverless: $40
→ EC2: $30
→ 超出：33%

70-100% 使用率：
→ Serverless: $60
→ EC2: $30
→ 超出：100%

轉換點：~50% 使用率
→ 以下：Serverless 較便宜
→ 以上：EC2 較便宜
```

## 真實案例研究

### 案例 1：Coca-Cola 的微服務

**挑戰：**
- 全球行銷活動管理
- 流量波動極大（促銷期間暴增 1000 倍）
- 需要快速部署新功能

**解決方案：**
```
從 300 個 monolithic 應用
  ↓
遷移到 Serverless 微服務
  ↓
使用 Lambda + API Gateway
```

**結果：**
- ✅ 開發時間從 6 個月降到 2 週
- ✅ 基礎設施成本降低 70%
- ✅ 自動處理 1000 倍流量峰值
- ✅ 運維人員減少 50%

### 案例 2：iRobot 的資料處理

**挑戰：**
- 處理數百萬台 Roomba 的數據
- 流量不可預測（用戶隨時清掃）
- 需要近乎實時的處理

**解決方案：**
```
每台 Roomba:
  ↓ 發送數據
[Kinesis Stream]
  ↓ 觸發
[Lambda 處理]
  ↓ 儲存
[S3 + Redshift]
```

**結果：**
- ✅ 處理 6PB 的數據
- ✅ 成本降低 60%
- ✅ 自動擴展處理峰值
- ✅ 零運維負擔

### 案例 3：哪裡不該用 Serverless

**挑戰：**
- 視訊會議平台
- 需要 <100ms 延遲
- 持續的連線

**錯誤決策：**
```
使用 Lambda 處理 WebRTC
  ↓
冷啟動延遲 500ms
  ↓
用戶體驗差
  ↓
流失用戶
```

**正確決策：**
```
使用 EC2 或 ECS
  ↓
固定實例，低延遲
  ↓
用戶體驗好
  ↓
成功
```
