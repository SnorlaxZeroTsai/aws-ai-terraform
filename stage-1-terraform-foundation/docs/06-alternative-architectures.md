# 替代架構方案

**學習目標：** 了解不同架構選項的取捨，知道何時該用哪種方案

---

## 回顧：我們目前的設計

在之前的章節我們學會了：
- VPC + 2 公有子網 + 2 私有子網
- NAT Gateway × 2
- EC2 實例
- 傳統 3 層架構

**這是 Stage 1 的基礎設計。**

## 為什麼要了解替代方案？

### 場景：需求改變

```
今天：100 個用戶
  → 使用 EC2 × 2 夠用

三個月後：10,000 個用戶
  → EC2 × 2 不夠了！
  → 需要重新設計架構

一年後：100,000 個用戶，全球分佈
  → 單一 Region 不夠了
  → 需要多 Region 架構
```

**問題：** 如果只知道一種方案，需求改變時就不知道該怎麼辦

**解決方案：** 了解各種架構的優缺點，知道何時該用哪種

## 架構分類光譜

### 計算服務的演進

```
┌─────────────────────────────────────────────────────────────┐
│ 託管程度 (從低到高)                                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  [裸機/VM] → [容器] → [Serverless] → [完全託管服務]         │
│                                                             │
│  你管理   你管理   AWS管理      AWS管理                     │
│  一切     OS      執行          一切                        │
│                                                             │
│  靈活    靈活     有限          靈活度最低                   │
│  成本高  成本中   成本低        成本不定                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 各階段的代表技術

| 架構類型 | 代表技術 | 你管理什麼 | AWS 管理什麼 |
|---------|---------|-----------|-------------|
| **裸機/VM** | EC2 | OS、Runtime、App、Scaling | 硬體、網路、資料中心 |
| **容器** | ECS/EKS | App、Container Image | Container Runtime、Scaling |
| **Serverless** | Lambda | Function Code | 執行環境、Scaling |
| **託管服務** | API Gateway | Configuration | 一切 |

## 方案 1：傳統 VM 架構（Stage 1 選擇）

### 架構圖

```
[Internet]
    ↓
[Application Load Balancer]
    ↓
[Auto Scaling Group: EC2 × N]
    ├─ EC2-1 (t3.medium)
    ├─ EC2-2 (t3.medium)
    └─ EC2-N (t3.medium)
    ↓
[RDS Database]
    ↓
[ElastiCache]
```

### 特點

**優點：**
- ✅ 完全控制環境
- ✅ 可預測的效能
- ✅ 支援所有語言和框架
- ✅ 長時間連線處理好
- ✅ 容易除錯

**缺點：**
- ❌ 需要管理 OS 更新
- ❌ 需要處理擴展
- ❌ 閒置時也要付錢
- ❌ 需要監控健康狀態

### 成本結構

```
固定成本：
- EC2 × 2 (24/7): $60/月
- ALB: $20/月
- NAT Gateway × 2: $64/月
- RDS: $135/月
- ElastiCache: $70/月
──────────────────────
小計：$349/月

變動成本：
- 資料傳輸：$0.01-0.045/GB
- EBS 儲存：$0.08/GB-month
- 備份：$0.05/GB-month

年固定成本：$4,188
```

### 適用場景

**✅ 適合使用：**
- 穩定的工作負載（CPU 使用率 > 50%）
- 需要長時間處理（如批次任務）
- 複雜的應用架構
- 需要特定 OS 設定
- 有專業的運維團隊

**❌ 不適合：**
- 流量波動大（< 10% 平均使用率）
- 無運維團隊
- 快速原型開發
- 間歇性任務

## 方案 2：容器架構

### 架構圖

```
[Internet]
    ↓
[Application Load Balancer]
    ↓
[ECS Cluster / EKS]
    ├─ Task Definition 1 × N
    ├─ Task Definition 2 × N
    └─ Task Definition N × N
    ↓
[RDS Database]
    ↓
[ElastiCache]
```

### 特點

**優點：**
- ✅ 更好的資源利用率
- ✅ 快速部署（秒級）
- ✅ 一致的執行環境
- ✅ 容易擴展
- ✅ 微服務友善

**缺點：**
- ❌ 需要容器化知識
- ❌ 需要管理 Container Registry
- ❌ 調試相對複雜
- ❌ 需要設計為無狀態

### 與 VM 的對比

| 特性 | VM | 容器 |
|------|-----|------|
| **啟動時間** | 分鐘 | 秒 |
| **資源利用率** | 40-60% | 70-90% |
| **隔離性** | 高 | 中 |
| **可移植性** | 低 | 高 |
| **學習曲線** | 中 | 高 |

### 成本結構

```
ECS (EC2 Launch Type):
- EC2 × 2 (t3.medium): $60/月
- ECR (Repository): $0.10/GB-month
- ALB: $20/月
──────────────────────
小計：$80/月

ECS (Fargate Launch Type):
- Fargate CPU: $0.04034/vCPU-hour
- Fargate Memory: $0.0044/GB-hour
- 假設 0.5 vCPU + 1GB × 2 任務，24/7:
  → CPU: 0.5 × 24 × 30 × $0.04034 = $14.5/月
  → Memory: 1 × 24 × 30 × $0.0044 = $3.2/月
  → 2 任務: ($14.5 + $3.2) × 2 = $35.4/月
- ALB: $20/月
──────────────────────
小計：$55/月

成本節省：Fargate vs EC2
→ 節省 ~25%（$55 vs $80，不包含 OS 管理成本）
```

### 適用場景

**✅ 適合使用：**
- 微服務架構
- 需要快速部署
- 多個小型服務
- CI/CD 自動化
- 開發/測試環境

**❌ 不適合：**
- 單體應用（難以拆分）
- 需要 OS 級別控制
- 複雜的網路設定

## 方案 3：Serverless 架構

### 架構圖

```
[Internet]
    ↓
[API Gateway]
    ↓
[Lambda Functions]
    ├─ auth-function
    ├─ analyze-function
    ├─ chat-function
    └─ process-function
    ↓
[Amazon DynamoDB / RDS Proxy]
    ↓
[S3 (for static assets)]
```

### 特點

**優點：**
- ✅ 無需管理伺服器
- ✅ 自動擴展
- ✅ 按使用付費（毫秒級）
- ✅ 高可用性內建
- ✅ 快速迭代

**缺點：**
- ❌ 執行時間限制（15 分鐘）
- ❌ 冷啟動延遲
- ❌ 不適合長時間任務
- ❌ 調試較困難
- ❌ 廠商鎖定風險

### 成本結構

```
Lambda:
- 免費額度：1M 請求/月 + 400,000 GB-秒
- 超過免費額度：
  → $0.20 per 1M 請求
  → $0.0000166667 per GB-second

假設場景：
- 100萬請求/月
- 每個請求執行 1 秒，記憶體 512MB
- 計算：
  → 請求費：$0.20
  → 執行費：1,000,000 × 1s × 0.5GB × $0.0000166667 = $8.33
  → 總計：$8.53/月

API Gateway:
- $3.50 per million requests
- 100萬請求：$3.50/月

DynamoDB:
- On-Demand: $1.25 per million write units
- On-Demand: $0.25 per million read units
- 假設 50萬寫入 + 50萬讀取：
  → 寫入：$0.63
  → 讀取：$0.13
  → 儲存（25GB）：$6.25
  → 總計：~$7/月

總成本：$8.53 + $3.50 + $7 = ~$19/月
```

### 與 VM 架構的成本對比

**場景：100 萬請求/月**

| 架構 | 月成本 | 年成本 | 閒置成本 |
|------|--------|--------|----------|
| **EC2 (2×t3.medium)** | $60 | $720 | $60（不管流量） |
| **Lambda** | $19 | $228 | $0（沒請求 = $0） |

**關鍵差異：**
- EC2：固定成本，不管有沒有流量
- Lambda：變動成本，完全根據使用量

### 適用場景

**✅ 適合使用：**
- 間歇性或不可預測的流量
- 事件驅動的架構
- 短時間任務（< 15 分鐘）
- API 服務
- 無運維團隊

**❌ 不適合：**
- 長時間運行的任務
- 高效能要求（低延遲）
- 需要特殊 OS 設定
- 持續的高流量（> 70% 時間）

## 方案 4：混合架構

### 架構圖：Serverless + Containers

```
[Internet]
    ↓
[CloudFront (CDN)]
    ↓
[API Gateway]
    ├─→ [Lambda: 輕量 API]
    │       ├─ auth
    │       ├─ validation
    │       └─ routing
    │
    └─→ [ECS/EKS: 重度任務]
            ├─ AI 模型推理
            ├─ 影片處理
            └─ 資料分析
    ↓
[Amazon Aurora Serverless v2]
    ↓
[ElastiCache Serverless]
```

### 設計原則

```
輕量、快速、無狀態 → Lambda
重度、長時間、有狀態 → ECS/EKS
靜態資源 → CloudFront + S3
資料庫 → Serverless（小流量）或 Provisioned（大流量）
```

### 成本最佳化

```
智能分流：
- 80% 流量：Lambda（便宜）
- 20% 重度任務：ECS（按需）

成本對比：
- 全 Lambda：$50/月（包含昂貴的長時間任務）
- 混合：Lambda ($30) + ECS ($20) = $50/月
- 全 ECS：$100/月（所有任務都付固定成本）

混合架構在複雜場景下最經濟
```

### 適用場景

**✅ 適合使用：**
- 多種類型的工作負載
- 需要最佳化成本
- 有一定的運維能力
- 成長中的產品

## 跨 Region 架構考慮

### 單 Region vs 多 Region

**單 Region：**
```
[us-east-1]
├─ 所有資源
└─ 延遲：全球用戶
```

**多 Region：**
```
[us-east-1] ← 北美用戶
[eu-west-1] ← 歐洲用戶
[ap-northeast-1] ← 亞洲用戶

透過 Route 53 做流量路由
```

### 成本比較

```
單 Region (us-east-1):
- EC2 × 4: $120/月
- RDS Multi-AZ: $270/月
- ALB: $20/月
────────────────
小計：$410/月

多 Region (3 Regions):
- EC2 × 4 × 3: $360/月
- RDS Multi-AZ × 3: $810/月
- ALB × 3: $60/月
- Route 53: $5/月
- 資料傳輸：$50/月
────────────────
小計：$1,285/月

成本增加：3.1 倍
```

### 什麼時候需要多 Region？

**✅ 需要的情況：**
- 全球用戶基礎（延遲要求）
- 合規要求（資料必須留在當地）
- 災難恢復需求
- 大型流量（> 1M 用戶）

**❌ 不需要：**
- 區域性服務
- 初期產品（< 100K 用戶）
- 預算有限

## 架構選擇決策框架

### 決策樹

```
開始
  ↓
有運維團隊？
  ├─ No → 考慮 Serverless 或託管服務
  └─ Yes → 繼續
      ↓
    流量穩定？
      ├─ Yes → 考慮 VM 或容器
      └─ No → 考慮 Serverless
          ↓
        任務執行時間？
          ├─ < 5 分鐘 → Serverless 完美
          ├─ 5-15 分鐘 → Serverless 或容器
          └─ > 15 分鐘 → 容器或 VM
              ↓
            需要控制 OS？
              ├─ Yes → VM
              └─ No → 容器
                  ↓
                微服務？
                  ├─ Yes → 容器
                  └─ No → VM 或 Serverless
```

### 成本與複雜度矩陣

```
           低複雜度  中複雜度  高複雜度
           ────────  ────────  ────────
低成本   Serverless  Serverless  容器
中成本   託管服務    Serverless  容器
高成本   VM         託管服務    VM
```

## 遷移路徑

### 從 VM 到容器

```
階段 1：評估
├─ 識別可容器化的應用
├─ 評估架構改動
└─ 成本效益分析

階段 2：準備
├─ 容器化應用（Dockerfile）
├─ 建置 ECR Repository
└─ CI/CD 調整

階段 3：遷移
├─ ECS Cluster 建立
├─ 任務定義設定
└─ 漸進式流量轉移

階段 4：優化
├─ Auto Scaling 調整
├─ 監控設置
└─ 成本最佳化
```

### 從 VM 到 Serverless

```
階段 1：重構
├─ 應用改為無狀態
├─ 拆分為小型函數
└─ 資料庫調整（DynamoDB 或 Proxy）

階段 2：實作
├─ Lambda 函數開發
├─ API Gateway 設定
└─ 測試和驗證

階段 3：遷移
├─ 平行運行（VM + Lambda）
├─ 流量逐步轉移
└─ 舊系統下線
```

### 什麼時候該考慮遷移？

**遷移觸發條件：**
- 流量增長 3 倍以上
- 成本增加 50% 以上
- 運維負擔過重
- 需要更快部署
- 團隊技能提升

## 常見錯誤

### 錯誤 1：過早優化

```
❌ 100 用戶就設計多 Region 架構
→ 浪費時間和金錢

✅ 從簡單開始，按需求成長
→ 100 用戶：單 Region Serverless
→ 10K 用戶：評估是否需要遷移
→ 1M 用戶：考慮多 Region
```

### 錯誤 2：廠商鎖定忽略

```
❌ 完全依賴 AWS 特定服務
→ 難以遷移到其他平台

✅ 保持可移植性
→ 使用標準技術（Docker）
→ 抽象化廠商特定功能
```

### 錯誤 3：忽視運維成本

```
❌ 只看直接成本
→ EC2 $100/月

✅ 考慮總擁有成本
→ EC2 $100/月
→ + 運維時間：10 小時/月
→ + 監控設置：5 小時/月
→ + 故障處理：不定
→ 真實成本 > $500/月（時薪 $50）
```
