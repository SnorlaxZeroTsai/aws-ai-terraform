**學習目標：** 理解如何設計 VPC，以及不同設計決策的權衡取捨

在上一章我們學到：
- ✅ VPC 提供網路隔離
- ✅ 完全的網路控制
- ✅ 多層安全防禦
- ✅ 符合合規要求

現在讓我們深入：**如何設計一個 VPC？**

## 第一個決策：VPC 的 IP 範圍

### 問題：VPC 應該用什麼 IP 範圍？

**VPC 的 IP 範圍決定了：**
- 這個 VPC 可以容納多少個 IP 地址
- 這些 IP 如何分配給子網
- 未來的擴展性

### 選項對比

| CIDR 範圍 | 可用 IP 數 | 適用場景 | 擴展性 |
|-----------|-----------|----------|--------|
| `/16` | 65,536 | 大型生產環境 | 最佳 |
| `/20` | 4,096 | 中型專案 | 良好 |
| `/24` | 256 | 小型測試 | 有限 |

### 為什麼選 /16？

**我們選擇 `10.0.0.0/16`**

**優點：**
- ✅ 65,536 個 IP 位址足夠使用
- ✅ 可以靈活劃分多個子網
- ✅ 未來擴展有餘裕
- ✅ 10.x.x.x 是私有範圍，不會與公網衝突

**缺點：**
- ❌ 對小型專案來說過大
- ❌ 浪費許多未使用的 IP

**其他常見選擇：**

```
172.16.0.0/12  → 1,048,576 IPs (太大，企業級)
192.168.0.0/16 → 65,536 IPs   (家用路由器常用)
10.x.x.x/16     → 65,536 IPs   (我們的選擇)
```

### 決策原則

**選擇 VPC 範圍時考慮：**

1. **未來需求：** 預留足夠空間給新服務
2. **子網規劃：** 每個子網需要一個 /24（256 IPs）
3. **跨 VPC 通訊：** 避免與其他 VPC 或資料庫 CIDR 衝突
4. **VPN 連線：** 如需連線到現有內部網路，避免衝突

**實例計算：**

```
假設需要：
- 2 個公有子網 (每 AZ 一個)
- 2 個私有子網 (每 AZ 一個)
- 2 個資料庫子網 (每 AZ 一個)
- 未來擴展：再預留 6 個子網

總共需要：8 個子網現在 + 6 個未來 = 14 個子網
每個子網 /24 = 256 IPs
總需求：14 × 256 = 3,584 IPs

/24 (256 IPs)     ❌ 不夠
/20 (4,096 IPs)   ✅ 勉強夠，但擴展性有限
/16 (65,536 IPs)  ✅ 最適合
```

## 第二個決策：子網設計

### 問題：如何劃分子網？

子網是 VPC 內的 IP 區段劃分，用於：
- 隔離不同類型的資源
- 控制流量流向
- 實施安全策略

### 子網類型對比

| 子網類型 | 是否有公網 IP | 用途 | 安全性 |
|---------|--------------|------|--------|
| **公有** | ✅ 有 | 對外服務 | 較低 |
| **私有** | ❌ 無 | 內部服務 | 較高 |

### 我們的設計：4 個子網

```
VPC: 10.0.0.0/16
│
├─ 公有子網 A (10.0.1.0/24)
│  └─ 用途：Load Balancer, API Gateway
│
├─ 公有子網 B (10.0.2.0/24)
│  └─ 用途：Load Balancer, API Gateway
│
├─ 私有子網 A (10.0.3.0/24)
│  └─ 用途：AI 應用伺服器
│
└─ 私有子網 B (10.0.4.0/24)
   └─ 用途：資料庫
```

### 為什麼這樣劃分？

**公有子網 (Public Subnets)**

**特徵：**
- 有公網 IP 位址
- 可以直接對外通訊
- 可以直接接收外部流量

**用途：**
```
[外部用戶] → [Internet Gateway] → [公有子網] → [Load Balancer]
                                                  ↓
                                            [公有子網] → [API 伺服器]
```

**放什麼資源：**
- ✅ Load Balancer（必須暴露）
- ✅ API Gateway（必須暴露）
- ✅ NAT Gateway（需要公網 IP）
- ✅ Reverse Proxy（Nginx, Envoy）

**不放什麼：**
- ❌ 資料庫（太危險）
- ❌ AI 模型服務（昂貴，不該暴露）
- ❌ 內部 API（不應暴露）

---

**私有子網 (Private Subnets)**

**特徵：**
- 沒有公網 IP
- 只能通過 NAT Gateway 或 VPC Endpoint 對外通訊
- 外部無法直接連線

**用途：**
```
[公有子網] → [安全群組規則] → [私有子網]
                                    ↓
                          [AI 應用伺服器]
                                [資料庫]
```

**放什麼資源：**
- ✅ 應用伺服器（EC2, ECS）
- ✅ AI 模型服務
- ✅ 資料庫（RDS, DynamoDB）
- ✅ 快取服務（ElastiCache）
- ✅ 內部 API

**為什麼這樣分離？**

**層級架構：**

```
[外部用戶]
    ↓
[DMZ - 公有子網] ← 第一道防線
    ↓ (經過驗證和過濾)
[應用層 - 私有子網] ← 第二道防線
    ↓
[資料層 - 私有子網] ← 最後防線
```

**好處：**
- ✅ 即使公有子網被攻破，私有子網仍是安全的
- ✅ 攻擊者無法直接存取資料庫
- ✅ 符合安全最佳實踐

---

### IP 分配實例

**我們的 VPC IP 分配：**

```
10.0.0.0/16 (VPC)
│
├─ 10.0.1.0/24 (公有子網 AZ-a)
│  ├─ 10.0.1.0 - 10.0.1.4:   AWS 保留
│  ├─ 10.0.1.5:              Load Balancer
│  ├─ 10.0.1.6 - 10.0.1.100: API 伺服器 (未來擴展)
│  └─ 10.0.1.101 - 10.0.1.250: 可用
│
├─ 10.0.2.0/24 (公有子網 AZ-b)
│  └─ (類似結構)
│
├─ 10.0.3.0/24 (私有子網 AZ-a)
│  ├─ 10.0.3.5:              AI 應用伺服器
│  ├─ 10.0.3.6 - 10.0.3.50:  未來應用
│  └─ 10.0.3.51 - 10.0.3.250: 可用
│
└─ 10.0.4.0/24 (私有子網 AZ-b)
   ├─ 10.0.4.5:              資料庫
   └─ (類似結構)
```

**為什麼每個子網用 /24？**

- 256 個 IP 位址
- 前 5 個 AWS 保留
- 剩下 251 個可用
- 對一個子網來說足夠了

## 第三個決策：單 AZ vs 多 AZ

### 問題：需要部署在幾個 Availability Zone？

**AZ（Availability Zone）= 數據中心內的獨立設施**

### 單 AZ 架構

```
[AZ-a]
│
└─ [所有資源]
   ├─ 公有子網
   ├─ 私有子網
   └─ 資料庫

成本：最低
風險：AZ 故障 = 服務中斷
```

**優點：**
- ✅ 成本最低（資源不重複）
- ✅ 設置簡單
- ✅ 排查問題容易

**缺點：**
- ❌ 單點故障
- ❌ AZ 維護時服務中斷
- ❌ 無法實現高可用性

**適用場景：**
- 開發/測試環境
- 原型驗證
- 非關鍵服務

---

### 多 AZ 架構（我們選擇）

```
[AZ-a]          [AZ-b]
  ↓               ↓
[公有子網]     [公有子網]
  ↓               ↓
[私有子網]     [私有子網]
  ↓               ↓
[資料庫]       [資料庫]
  ↓               ↓
  └─ [同步] ────┘

成本：×2（資源重複）
風險：AZ 故障時另一個 AZ 繼續服務
```

**優點：**
- ✅ 高可用性
- ✅ 災難恢復
- ✅ 低延遲（就近服務）
- ✅ 符合生產標準

**缺點：**
- ❌ 成本翻倍（資料庫特別貴）
- ❌ 複雜度增加
- ❌ 跨 AZ 資料同步

**適用場景：**
- 生產環境
- 關鍵任務系統
- 需要高可用的服務

---

### 權衡分析：成本 vs. 可靠性

```
可靠性
  ↑
  │           [多 AZ]
  │          /
  │         /
  │        /  ← 增加可靠性
  │       /
  │      /
  │     /
  │────/
  └──────────────→ 成本
   [單 AZ]
```

**量化比較：**

| 項目 | 單 AZ | 2 AZ |
|------|-------|------|
| **月成本** | $25 | $50 |
| **可用性** | ~99.5% | ~99.95% |
| **年度停機** | ~18 小時 | ~4 小時 |
| **災難恢復** | 手動 | 自動 |

**什麼時候選多 AZ？**

| 情境 | 單 AZ | 多 AZ |
|------|-------|------|
| 開發環境 | ✅ | ❌ 過度 |
| 測試環境 | ✅ | ❌ 成本高 |
| Staging | ❌ | ✅ |
| 生產 | ❌ | ✅ 必須 |

**我們選擇 2 AZ 的原因：**
- 學習生產級架構
- 了解高可用設計
- 成本可控（只用小實例）

## 第四個決策：路由策略

### 問題：流量如何在 VPC 內流動？

**路由表 = 流量 GPS**
- 決定流量往哪裡走
- 每個子網必須有路由表
- 公有和私有子網有不同的路由策略

### 公有子網路由表

```
路由表：public-rt
├─ 10.0.0.0/16 → local (VPC 內部通訊)
└─ 0.0.0.0/0   → Internet Gateway (對外通訊)

流量路徑：
[公有子網的 EC2] → [Internet Gateway] → [外部 API]
                    ↑
                 (可以直接上網)
```

**為什麼這樣設計？**
- 公有子網需要對外服務
- Load Balancer 需要接收外部請求
- NAT Gateway 需要公網 IP

---

### 私有子網路由表

```
路由表：private-rt
├─ 10.0.0.0/16 → local (VPC 內部通訊)
└─ 0.0.0.0/0   → NAT Gateway (透過 NAT 上網)

流量路徑：
[私有子網的 EC2] → [NAT Gateway] → [Internet Gateway] → [外部 API]
                        ↑
                   (不能直接上網，要經過 NAT)
```

**為什麼要這樣？**
- 私有子網沒有公網 IP
- 需要經過 NAT 才能對外通訊
- 保護私有資源不被直接存取

## VPC 設計最佳實踐

### 1. 預留 IP 空間

**不要一次用完所有 IP：**

```
錯誤：
10.0.0.0/16
├─ 10.0.1.0/24  (用完了)
├─ 10.0.2.0/24  (用完了)
├─ 10.0.3.0/24  (用完了)
└─ 10.0.4.0/24  (用完了)
→ 無法擴展

正確：
10.0.0.0/16
├─ 10.0.1.0/24  (現在使用)
├─ 10.0.2.0/24  (現在使用)
├─ 10.0.3.0/24  (現在使用)
├─ 10.0.4.0/24  (現在使用)
├─ 10.0.5.0/24 - 10.0.255.0/24  (保留給未來)
```

### 2. 命名規範

**使用有意義的命名：**

```
10.0.1.0/24 → public-subnet-az-a
10.0.2.0/24 → public-subnet-az-b
10.0.3.0/24 → private-app-subnet-az-a
10.0.4.0/24 → private-db-subnet-az-a
```

**好處：**
- ✅ 一眼看出用途
- ✅ 便於管理
- ✅ 降低錯誤機率

### 3. 文檔化

**記錄你的設計決策：**

```markdown
# VPC 設計文檔

## VPC 規劃
- CIDR: 10.0.0.0/16
- AZ: us-east-1a, us-east-1b

## 子網分配
- 公有子網: 10.0.1.0/24, 10.0.2.0/24
- 應用子網: 10.0.3.0/24, 10.0.4.0/24
- 資料子網: 10.0.5.0/24, 10.0.6.0/24

## 路由策略
- 公有子網 → Internet Gateway
- 私有子網 → NAT Gateway
```

## 常見錯誤

### 錯誤 1：VPC 太小

**問題：**
```
使用 /24 當 VPC CIDR
→ 只有 256 個 IP
→ 只能劃分 2-3 個子網
→ 未來無法擴展
```

**修正：**
```
使用 /16 或至少 /20
→ 至少 4,096 個 IP
→ 可以劃分 16 個子網
→ 有擴展空間
```

### 錯誤 2：所有資源放公有子網

**問題：**
```
[資料庫] ←→ [公網]
[應用] ←→ [公網]
→ 完全暴露在攻擊下
```

**修正：**
```
[資料庫] ←→ [私有子網]
[應用]   ←→ [公有子網]
→ 多層防禦
```

### 錯誤 3：跨 AZ 的資料庫延遲

**問題：**
```
資料庫主節點在 AZ-a
資料庫從節點在 AZ-b
→ 跨 AZ 複製延遲 ~5-10ms
→ 影響寫入效能
```

**考量：**
- 如果延遲敏感，考慮單 AZ + 備份
- 如果可用性優先，接受延遲成本

## 實際案例：為什麼這樣設計？

### 案例：E-commerce 網站

**需求：**
- 必須 24/7 運作
- 流量波動大
- 促銷期間流量暴增 10 倍
- 資料庫不能停機

**VPC 設計：**

```
VPC: 10.0.0.0/16

公有子網 (3 AZ):
├─ 10.0.1.0/24 (AZ-a) → Load Balancer
├─ 10.0.2.0/24 (AZ-b) → Load Balancer
└─ 10.0.3.0/24 (AZ-c) → Load Balancer

應用子網 (3 AZ):
├─ 10.0.11.0/24 (AZ-a) → 應用伺服器
├─ 10.0.12.0/24 (AZ-b) → 應用伺服器
└─ 10.0.13.0/24 (AZ-c) → 應用伺服器

資料子網 (3 AZ):
├─ 10.0.21.0/24 (AZ-a) → 資料庫主
├─ 10.0.22.0/24 (AZ-b) → 資料庫從
└─ 10.0.23.0/24 (AZ-c) → 資料庫從

預留：10.0.100.0/24 - 10.0.255.0/24
```

**為什麼 3 AZ？**
- 促銷期間流量大，需要更多容量
- 任何單一 AZ 故障不影響服務
- 合規要求（金融業標準）

**成本：**
- 月成本約 $200-300
- 但停機損失可能數萬
- 保險值得
